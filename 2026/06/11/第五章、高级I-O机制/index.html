系统编程：高级 I/O 机制

高级 I/O 主要解决阻塞、复用、映射、锁和管道等场景下的效率与同步问题。

一、高级 I/O 解决什么问题

普通文件读写只需要 read() / write() 就能完成，但真实系统里经常会遇到更复杂的问题：

  • 一个程序要同时处理多个文件描述符。
  • 某些 I/O 操作可能长期阻塞。
  • 需要把文件直接映射到内存操作。
  • 多进程需要协调访问同一个文件。
  • 进程之间需要通过管道传递数据。

这就是高级 I/O 的意义。它不是替代基础 I/O，而是在基础 I/O 之上解决效率、并发和同步问题。

二、非阻塞 I/O

非阻塞 I/O 的三种理解方式：

  1. 将文件描述符设置为非阻塞。
  2. 让系统调用以非阻塞方式工作。
  3. 配合 select、poll、epoll 等 I/O 多路复用机制。

常见设置方式是通过 fcntl() 给 fd 加上 O_NONBLOCK：

非阻塞并不是“更快”的同义词。它的核心是：如果当前不能完成操作，调用立即返回，而不是让线程睡眠等待。

典型返回情况是 read() 或 write() 返回 -1，并设置 errno 为 EAGAIN 或 EWOULDBLOCK。程序需要识别这类情况，稍后再试。

  • 正在读取请求头。
  • 正在读取请求体。
  • 正在处理数据。
  • 正在写响应。
  • 写了一部分，等待下次继续写。

这就是状态机。网络服务器、代理、中继程序、终端交互程序都会大量使用这种思想。

四、I/O 多路转接：select

select() 的总结：

  • 以事件为单位组织文件描述符。
  • 可移植，支持多种操作系统。
  • 监视文件描述符数量有限，通常受 FD_SETSIZE 限制。
  • 每次调用都要把集合从用户空间复制到内核空间，效率较低。
  • 监视现场和结果保存在同一块空间，调用后原集合会被修改。

select() 的典型问题是：每次调用前都要重新初始化 fd 集合。否则上一次返回结果会覆盖原监控集合。

它适合学习 I/O 复用原理，也适合 fd 数量不多、跨平台要求高的场景。

五、poll 与 epoll

  • poll() 可移植，解决了 select() 监视 fd 数量有限的问题。
  • poll() 以文件描述符为单位组织事件。
  • epoll() 是 Linux 特有机制，不可移植，但性能更好。

可以简单对比：

机制 特点 适合场景
select 可移植，fd 数量有限 小规模、多平台
poll 可移植，数组组织 fd 中等规模
epoll Linux 特有，事件驱动效率高 大量连接、高并发

epoll 的优势不是“任何情况下都更快”，而是在大量 fd 中只有少量活跃时，可以避免反复线性扫描和大量数据复制。

七、存储映射 I/O：mmap

mmap() 可以把文件映射到进程地址空间。映射成功后，程序像访问内存一样访问文件内容。

典型用途：

  • 大文件随机访问。
  • 进程间共享内存。
  • 文件内容快速扫描。
  • 匿名映射作为特殊内存分配方式。

提醒：如果 mmap() 的 flags 参数使用匿名映射，那么 mmap() / munmap() 可以在某些场景下类似 malloc() / free() 使用。

需要注意：

  • 映射长度通常要按页对齐处理。
  • 写映射文件时，要理解 MAP_SHARED 和 MAP_PRIVATE 的区别。
  • 文件被截断后继续访问原映射区域可能触发 SIGBUS。

八、文件锁

三类文件锁接口：

  • fcntl()
  • lockf()
  • flock()

文件锁用于协调多个进程对同一文件的访问。典型场景是日志、数据库文件、状态文件、单实例程序。

“意外解锁问题”：close() 会自动释放锁。更细一点说，传统记录锁和进程、inode 关联密切，多个 fd 指向同一个文件时，关闭其中一个 fd 可能影响该进程持有的锁。这类语义比较容易出错，写程序时要明确锁的生命周期和 fd 的生命周期。

九、管道与读者写者问题

管道“类似于队列”。这是很好的直觉。管道是一种先进先出的字节流通信机制，常见于父子进程之间，也常见于 shell 命令组合：

管道有读端和写端。读写双方是否存在，会影响阻塞和返回行为：

  • 没有数据且写端未关闭，读端可能阻塞。
  • 写端关闭后，读端读完剩余数据再读会返回 0。
  • 读端关闭后，写端继续写可能收到 SIGPIPE。

读者写者问题则是同步模型中的经典问题：多个读者可以并发读，但写者需要独占访问。读写锁正是这个问题的常见解决方式。

十、巩固任务

  1. 把标准输入设置为非阻塞，观察 EAGAIN。
  2. 用 select() 同时监听标准输入和管道。
  3. 用 poll() 改写同样功能。
  4. 在 Linux 上用 epoll() 写一个简单 echo server。
  5. 用 mmap() 实现文件复制或关键字扫描。

总结

高级 I/O 的核心不是 API 数量，而是 I/O 模型的变化。阻塞、非阻塞、多路复用、内存映射和文件锁分别解决不同层面的问题。真正写程序时，要先判断瓶颈和需求，再选择机制：小规模用简单模型，大量连接用事件驱动，共享文件用锁，大文件随机访问考虑 mmap()。
